## 1. 운영 제약 (반드시 지킬 것)

1. **무중단·무오류.** 실사용자 있음. 미들웨어/RLS/인증 흐름은 깨지면 사이트 전체가 죽는다.
2. **워크플로**: `git checkout -b feat/xxx` → 빌드(`npm run build` = `tsc -b && vite build`) → **preview 배포에서 병원/의료인 각 계정으로 수동 검증** → main 머지 → 푸시.
3. **사용자 개입이 필요한 것 (세션 중 승인/실행 못 함 전제로 설계)**:
   - **Supabase SQL / 대시보드 작업**: 사용자가 콘솔에서 직접 실행해야 함. → 코드에서 assume 하지 말고, **바로 복붙 실행 가능한 SQL + 검증 쿼리 + 롤백**을 문서로 패키징해서 넘길 것.
   - **Supabase Edge Function 배포**, 환경변수, 도메인/키 회전, 결제·외부 계정.
   - 즉, 사용자는 "읽고 승인/실행"만. 그러니 **작업을 배치로 묶고, 사람이 할 일은 체크리스트로 분리**해서 제시.
4. **RLS가 유일한 보안 계층.** 백엔드가 없어 브라우저가 anon 키로 직접 Supabase 호출 → 모든 접근 통제는 Postgres RLS/뷰뿐. 새 테이블·컬럼 추가 시 RLS부터 설계.
5. **지도 핵심 규칙 (건드리지 말 것)**: **병원 = 정확한 위치 / 의료인 = 흐린 위치**(`jitterCoords` ±330m, `formatDistance` 버킷 표시). PII·정확좌표는 원본 노출 금지.

**사용 가능한 서브에이전트**: `pm-medinoti`(유저스토리·스코핑), `dev-medinoti`(구현), `marketer-medinoti`(카피·아웃리치). 큰 작업은 pm → dev 순서.

## 2. 현재 구조 (사실 관계)

- **페이지**(`src/pages/`): Landing, Login, Dashboard(지도·검색 핵심), RegisterWorker/EditProfile, RegisterHospital/EditHospital, PostJob/EditJob/MyJobs, PrivacyPolicy, TermsOfService.
- **컴포넌트**: Footer, LanguageSwitcher, PrivacyConsent(가입 동의), RouteGuard(역할 가드).
- **lib**: supabase, geocode, distance(흐림·거리버킷), profileFields, medicalConstants(치과 우선 정렬), jobPostingDisplay, sanitize(URL/tel 화이트리스트).
- **DB 테이블**: `profiles`(의료인+병원, role/license_type로 구분), `job_postings`, `public_profiles`(안전 컬럼만 노출하는 뷰). 스키마는 단순.
- **연결 방식**: 병원이 의료인/공고를 보고 **전화(getEffectiveMobile: 공고전용번호 우선) 또는 카카오 링크**로 직접 연락. 제3자 제공 동의한 의료인만 연락처 노출. **인앱 메시징/알림은 없음.**
- **i18n**: `detectLanguage()` 우선순위 = localStorage `mn_lang`(수동) > `mn_country` 쿠키(IP, `middleware.ts`) > navigator.language > ko. 한=ko/일=ja/그외=en. 법적 페이지는 한국어 유지.

## 4. 전략 우선순위 (핵심 — 여기서 다음 방향을 정한다)

> 프레임: 2-sided 마켓플레이스의 성패는 **① 유동성(liquidity) ② 측정 ③ 재참여**에서 갈린다.
> 현재 이 셋이 모두 약하다. 아래는 **임팩트/난이도** 기준 추천 순.

### 🥇 A. 측정 인프라부터 (제일 싸고, 나머지 결정의 전제)
**문제**: 분석 도구가 하나도 없다. 방문·가입·이탈·매칭·외국 트래픽을 아무도 모른다. i18n·광고를 해도 효과를 못 잰다.
**할 일**: 프라이버시 친화 분석(예: Plausible/Umami 자가호스팅 또는 Vercel Analytics) + 핵심 퍼널 이벤트(방문→가입 시작→가입 완료→공고 조회→연락처 클릭) 정의. 개인정보·의료광고법 고려해 PII 미수집 설정.
**왜 먼저**: B·C의 우선순위와 i18n ROI를 데이터로 판단하려면 필수. 반나절 작업.
**주의**: 쿠키/추적은 개인정보처리방침 반영 필요. 한국 대상이면 최소수집.

### 🥈 B. 재참여 루프 (마켓플레이스 리텐션 엔진)
**문제**: 가입 후 "나에게 맞는 새 공고/인재"를 알려주는 통보가 없다. 한 번 보고 이탈하면 끝. 마켓플레이스에서 매칭 알림은 리텐션의 심장.
**할 일**: Supabase **Edge Function + pg_cron**으로 "당신 근처/직군의 새 공고" **주간 다이제스트 이메일**(의료인용) / "새 지원 가능 인재"(병원용). 이메일 발송은 Resend 등.
**정합성**: 사용자 글로벌 계획의 "아침 다이제스트 봇/텔레그램" 방향과 일치. 백엔드 없이 가능한 유일한 재참여 수단.
**주의**: 광고성 정보 수신 동의(가입 동의에 이미 제3자 항목 있음 — 마케팅 수신 동의 별도 확인), 발송 도메인 인증(SPF/DKIM).

### 🥉 C. 유동성/콜드스타트 — "어느 쪽을 먼저 채울지" 결정
**문제**: 병원(수요)과 의료인(공급) 중 **희소한 쪽이 자석**. 한국 **치과위생사는 만성 공급 부족** → 보통 **공급(의료인)이 희소**. 광고를 "치과 집중"으로 이미 잡았으니, **의료인 풀을 먼저 씨딩 → 병원은 쉽게 전환**되는 전략이 유력.
**할 일(가설)**: (1) 치과위생사·치과 직군 의료인 획득 캠페인(marketer-medinoti). (2) 가입 퍼널 단축 — 현재 가입 폼 필드가 76~78개로 김 → 이탈 위험. **최소 가입 → 나중에 프로필 보강** 구조 검토(pm-medinoti). (3) 지역 밀도 전략: 한 지역(예: 인천/수도권 일부)부터 밀도를 채워 매칭이 실제로 일어나게.
**검증 먼저**: A의 데이터로 "어느 쪽이 더 잘 들어오고 덜 빠지는지" 본 뒤 확정.

### 향후 큰 베팅 (데이터 확인 후)
- **신뢰 계층**: 면허/사업자 **검증 배지**. 현재 자기신고 → 검증이 차별화·전환율 핵심이 될 수 있음.
- **SEO/발견성**: SPA라 organic 약함. "치과위생사 구인/구직 [지역]" 랜딩(SSG/프리렌더)로 유입. 단, 사용자 WordPress GEO(병원용)와 리소스 경쟁 — 우선순위 낮게.
- **i18n ROI 검증**: EN/JA는 깔았으나 실제 외국 수요는 미지수. A의 국가별 트래픽으로 **추가 투자 여부 판단**. 수요 없으면 유지만 하고 확장 보류.

## 5. 리스크 / 워치아웃

- **미들웨어 = 단일 장애점**: `middleware.ts` 런타임 에러 시 매칭 경로 전체 500. 이미 try/catch로 방어했으나, 수정 시 반드시 preview 검증.
- **RLS 실수 = 데이터 유출/차단**: 새 쿼리·테이블은 RLS부터. 특히 의료인 PII·정확좌표 원본이 네트워크 응답에 새지 않는지 Network 탭 확인(C-1/C-2 원칙).
- **가입 폼 이탈**: 필드 과다. 측정 없이 손대지 말고, A 이후 데이터로.
- **의료광고법(B2C 성격 콘텐츠)**: 마케팅 카피는 과장·비교·보장 금지. 글로벌 CLAUDE.md 규칙 + `medical-ad-law-auditor` 검수.
- **개인정보/추적**: 분석·이메일 도입 시 개인정보처리방침 갱신 동반.
